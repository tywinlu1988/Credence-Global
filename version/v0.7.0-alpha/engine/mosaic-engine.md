# 马赛克引擎

**版本**: v0.7.0-alpha | **日期**: 2026-07-10
**核心原则**: 承认数据永远残缺，把"残缺"从弱点变成产品特征

---

## 一、核心理念

### 1.1 两个基础理论

| 理论 | 含义 | 在引擎中的体现 |
|---|---|---|
| **马赛克理论** | 单块公开数据碎片无意义，拼在一起构成完整画像。每一片公开信息——政策文件、法院公告、招标结果、价格周报——单独看都不能说明问题，但聚合后形成判断 | 多源数据聚合 → 信号叠加 → 置信度加权 |
| **信息完备性理论** | 不仅回答"知道了什么"，更回答"还不知道什么"。信息缺口不是缺陷，而是风险信号的来源——"我们没有这个数据"本身就是在告诉用户"这个维度存在不确定性" | 每个分析结论附带数据完备性评分和缺口列表 |

### 1.2 马赛克引擎在整体架构中的位置

```
技能包 v0.3.0
  ├── 行业分类（十维评分）
  ├── 轨道A：基本面分析（金字塔框架）
  ├── 轨道B：市场定价（四级信号）
  ├── 交叉对撞（四象限矩阵）
  ├── 黑天鹅回溯验证
  │
  └── 马赛克引擎层 ← 本文档
        ├── 信号提取层
        ├── 马赛克拼图层
        ├── 完备性评估层
        └── 模式B接口定义（占位）
```

## 三、信号提取层

### 3.1 功能

**输入**：WebSearch返回的搜索结果（标题+摘要+URL）
**输出**：结构化信号对象

### 3.2 信号数据结构

```yaml
signal:
  id: "sig-001"
  source_url: "https://..."
  source_type: "policy_document" | "court_record" | "bidding_result" | "industry_report" | "news"
  timestamp: "2026-06-15"  # 信息来源的时间，不是抓取时间
  dimension: "L2_technology"  # 属于金字塔哪一层
  content: "隆基BC电池量产效率达24.8%"
  direction: "positive" | "negative" | "neutral"
  strength: 1-5  # 1=弱方向性信号, 5=明确量化信号
  cross_validated: false  # 进入拼图层后若有≥2独立源确认则改为true
```

### 3.3 提取规则

- LLM从每条搜索结果的标题+摘要中提取信号
- **不做判断，只做结构化**
- 最多返回3条信号/源
- 每片数据都标注其所属的信号置信度层级

## 四、马赛克拼图层

### 4.1 核心逻辑

```
对于每个分析维度（如L1政策环境）：
  1. 汇总该维度的所有信号
  2. 同方向信号叠加 → 增加置信度
  3. 矛盾信号出现 → 标注"分歧"并降低置信度
  4. 跨源交叉验证 → 提升信号层级
  5. 计算该维度的"信号密度" = 有效信号数 / 该维度应有信号数的预估
```

### 4.2 五级信号置信度

| 级别 | 类型 | 含义 |
|---|---|---|
| **L5** | 多源交叉验证信号 | 同一事实被≥2个独立来源确认 |
| **L4** | 单源直接信号 | 一个可靠来源明确陈述 |
| **L3** | 衍生推断信号 | 从多片碎片组合推理得出 |
| **L2** | 方向性弱信号 | 只有方向，没有量级 |
| **L1** | 缺失信号（缺口） | 应该有的数据没有——风险信号 |

**层级优先级**：L5多源交叉验证 > L4单源直接信号 > L3衍生推断 > L2方向性弱信号 > L1缺口（单独标注）。

### 4.3 信号密度指标

信号密度是马赛克拼图层**最关键的输出**——直接回答"我们对这个维度了解多少"。

| 信号密度 | 含义 | 对最终评分的影响 |
|---|---|---|
| **>80%** | 信息充分——关键信号基本齐全 | 高置信度，可直接给出评分 |
| **50-80%** | 信息中等——主要信号有但细节不足 | 中置信度，评分附带±1的误差区间 |
| **20-50%** | 信息不足——只有碎片 | 低置信度，标注为"待补充"，仅给出方向性判断 |
| **<20%** | 信息严重缺失 | 不给出该维度评分，标注为"信息不足无法评估" |

## 五、完备性评估层（核心差异化模块）

这是产品差异化的关键。不仅告诉用户"我们看到了什么"，更告诉用户"我们没看到什么，这意味着什么风险"。

### 5.1 每个维度的三要素输出

```
维度：L2 技术路线与竞争位势
├── 信号密度：75%（信息中等）
├── 已获取的关键信号：
│   ├── ✅ 组件量产效率（隆基BC 24.8% vs TOPCon ~23.5%）
│   ├── ✅ 央企集采入围情况（2026年集采59GW数据）
│   ├── ✅ 专利数量（隆基510件 vs 晶科/天合）
│   └── ⚠️ 产能利用率（行业估算值，非企业披露）
├── 缺失的信号：
│   ├── ❌ 各企业的良率数据（企业不单独披露）
│   ├── ❌ 各技术路线的非硅成本精确对比（仅公开估算）
│   └── ❌ BC vs TOPCon在分布式场景的实际发电量对比（无第三方测试）
└── 缺口风险评估：
    → 良率数据缺失 → 无法精确判断成本竞争力 → 评分区间 ±1.5
    → 但央企集采入围数据和效率差距已构成足够的区分信号
```

### 5.2 数据缺口→风险映射表（通用模板）

| 缺口类型 | 典型缺失数据 | 对应的信息风险 | 替代信号 |
|---|---|---|---|
| 竞争数据 | 各企业成本精确对比 | 高估或低估成本优势 | 央企集采中标价（市场接受的溢价） |
| 财务数据 | 非上市企业无公开财报 | 完全无法评估财务健康 | 被执行记录、司法纠纷、招聘趋势 |
| 市场定价 | 非上市企业无股价/债券 | 轨道B完全不可用 | 公开融资事件、股权转让价格 |
| 条款数据 | 无Wind终端无法获取久期/Z-spread | 无法做精细利率风险分析 | YTM近似、同评级利差对比 |
| 流动性数据 | Bid-ask spread在中国不披露 | 无法评估真实交易成本 | 日均成交量、换手率、异常成交量事件 |
| 外部支持 | 地方政府隐性债务真实规模 | 高估或低估政府真实支持能力 | 显性债务率+城投有息负债估算，标注"隐性债务估算误差" |
| 外部支持 | 政府对特定企业的救助承诺（非公开） | 高估支持意愿的确定性 | 历史支持记录+股权结构+战略定位推断，标注"非正式承诺" |
| 外部支持 | 集团内部资金归集政策和实际执行情况 | 无法判断集团对子公司的实际资金支持力度 | 关联方往来款+担保余额+业务协同度，标注"基于公开信息推断" |
| 外部支持 | 战略投资者实际支持意愿（非公开表态） | 高估战投支持的可执行性 | 投资协议条款的公开披露+锁定期+董事会构成 |

### 5.3 完备性评估报告结构

每个分析的输出**必须**包含以下结构：

```
1. 综合评级与核心发现
   └── 置信度标注（高/中/低）

2. 数据完备性报告（信号密度条形图）
   ┌─────────────────────────────────────┐
   │ L1 政策/宏观    ████████░░ 82%      │
   │ L2 技术/竞争    ██████░░░░ 75%      │
   │ L3 供应链/运营  ████░░░░░░ 48% ⚠️   │
   │ L4 财务/偿债    █████████░ 89%      │
   │ L5 外部支持    ██████░░░░ 60% ⚠️   │
   │ 市场定价(轨B)   ███░░░░░░░ 35% ⚠️   │
   │ 相对价值(M1.1)  ██░░░░░░░░ 25% ❌   │
   │ 条款分析(M1.2)  ████████░░ 80%      │
   │ 流动性(M1.3)    ████░░░░░░ 45% ⚠️   │
   └─────────────────────────────────────┘

3. 缺口清单（按影响优先级排序）
   └── 每个缺口包含：具体缺失数据 → 影响 → 替代方案

4. 评分置信区间
   └── 基于信号密度对各维度的评分区间做修正

5. 用户可操作建议
   └── "如需精细相对价值分析，建议接入Wind/Choice数据终端"
```

### 5.4 置信度对评分的影响规则

| 信号密度 | 评分输出规则 |
|---|---|
| >80% | 直接输出精确评分 |
| 50-80% | 输出评分 ±1 区间 |
| 20-50% | 仅输出方向性判断（正面/负面/中性），不输出精确评分 |
| <20% | 标注"信息不足无法评估"，不给出该维度评分 |

标准化的接口定义，允许用户连接自己的数据源补全马赛克缺口。已定义契约但未实现。

```yaml
DataSourceAdapter:
  # 生命周期
  init(config):        # 认证、限流配置

  # 核心查询
  query_bond_analytics(bond_code, fields):
    # fields: ["ytm","duration","convexity","z_spread","oas","bid_ask"]
    # 返回: {field: value, ...} + data_timestamp

  query_market_data(instrument_code, data_type, date_range):
    # data_type: "price_history" | "volume" | "volatility" | "fund_flow"

  query_industry_benchmark(industry, metric, date):
    # metric: "credit_spread" | "default_rate" | "rating_migration"

  to_signals():  # 将外部数据转换为统一的信号对象格式
```

### 6.2 支持的接入方式（规划）

| 接入方式 | 适用场景 | 优先级 |
|---|---|---|
| REST API | Wind/Choice/同花顺等金融终端API | P1 |
| MCP Server | 用户自建的数据服务 | P2 |
| CSV/Excel上传 | 一次性分析，用户手动导出数据 | P1 |
| 数据库直连 | 用户内部数据仓库 | P3 |

### 6.3 模式B数据流

```
用户提供API Key / MCP Endpoint
         │
    ┌────┴────┐
    │ 数据适配层│  ← 标准化接口：认证/限流/缓存
    └────┬────┘
         │
    ┌────┴────┐
    │ 信号增强层│  ← 外部数据→信号→与模式A的信号融合
    │ 补全缺口  │  ← 外部数据填补的缺口自动标记为"已补全"
    └────┬────┘
         │
         ↓ 汇入马赛克拼图层（与模式A相同的处理流程）
```

## 七、实例：马赛克理论如何工作

### 案例：判断通威股份的偿债能力

**五片碎片——每片单独看都不够：**

| 碎片 | 信号 | 单独看的问题 |
|---|---|---|
| 通威短融2025年利率从2.60%↓2.02%（-58bp） | 市场认为风险在降低 | 可能只是货币环境宽松 |
| 通威短融2026年1月利率从2.02%↑2.15%（+13bp） | 市场开始重新定价风险 | 反弹幅度太小，可能只是噪声 |
| 通威2025年亏损90-100亿 | 基本面严重恶化 | 亏损可能是一次性的（硅料价格暴跌） |
| 通威应收账款从Q3跳升至Q4 | 回款能力在恶化 | 可能只是季节性因素 |
| AAA/稳定评级维持 | 评级机构认为风险可控 | 评级滞后是已知问题 |

**拼在一起后的完整画像：**

```
碎片1（利差压缩） + 碎片2（利差反弹） = 利差拐点信号（弱→中等强度）
碎片2（利差反弹） + 碎片3（巨额亏损） = 市场开始反映基本面（强度升级）
碎片3（亏损） + 碎片4（应收跳升）     = 不仅亏损，回款也在恶化（双杀信号）
碎片5（AAA维持） + 碎片1-4            = 评级滞后确认（评级信号不可信）

马赛克拼图结论：
通威的偿债能力在恶化，市场定价（利差反弹）开始反映这一趋势，
但AAA评级仍在滞后。

信息完备性评估：中等
缺口：通威母公司独立财务数据不可获取
→ 无法精确判断母公司层面的真实偿债能力
```

### 马赛克拼图的过程演示

| 步骤 | 操作 | 结果 |
|---|---|---|
| 1 | 提取5条原始信号 | 各信号单独来看置信度均为L2-L4 |
| 2 | 同方向信号叠加 | 利差先降后升 → 拐点信号（L3→L4升级） |
| 3 | 跨维度交叉验证 | 亏损（L3）+ 应收跳升（L3）= 双杀（L4→L5强制降级） |
| 4 | 矛盾信号处理 | AAA评级 vs 其他4条信号 → 评级信号不可信 |
| 5 | 信号密度计算 | 5维度中有4条可用信号 → 80%（中等偏上） |
| 6 | 缺口识别 | 母公司财务数据缺失 → 影响L4评分置信度±1 |

## 八、实施优先级

| 优先级 | 模块 | 说明 |
|---|---|---|
| **P0** | 信号提取层 + 马赛克拼图层 | 当前P0/P1/P2所有分析的基础 |
| **P0** | 完备性评估层 | **核心差异化**——缺口标注引擎 |
| **P1** | 缺口→风险映射表（按行业定制） | 每个行业的缺失数据对应什么风险 |
| **P1** | 模式B：CSV上传适配器 | 最低成本的"外部数据"入口 |
| **P2** | 模式B：REST API适配器 | Wind/Choice等专业终端 |
| **P3** | 模式B：MCP适配器 | 用户自建数据服务 |
