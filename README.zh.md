## 什么是 Credence

Credence 将资深固定收益信贷分析师的方法论打包为 AI agent 可直接加载执行的形态。它**不是 agent 框架，也不是独立应用**，而是一个**领域方法论技能包**，专为国际债券市场的机构级信用分析而设计。

### 核心理念

传统财务分析基于一个隐含假设——企业的信用风险可以从其财务报表中读取。这一假设在三种行业类型中系统性失效：

| 行业类型 | 财务分析失效的原因 | 最关键因素所在 |
|---|---|---|
| **政策驱动型**（光伏、半导体） | 政策周期决定行业需求天花板；政策突变可在数周内摧毁行业 | 产业政策/地缘政治（不在资产负债表上） |
| **技术壁垒型**（高端装备、生物医药） | 核心资产（知识产权、管线、认证）不在资产负债表上；许多收入前企业无法用 PE/PB 估值 | 技术路线图/核心知识产权（不在资产负债表上） |
| **资产租赁型**（数据中心、基础设施 REITs） | 类似 REIT；核心指标是 NOI/DSCR 而非传统财务指标 | 客户租赁质量（不在资产负债表上） |

**最关键的信用因素永远不会出现在资产负债表上。** 外部信用评级平均延迟真实信用恶化 17 个月以上（安然、雷曼兄弟、Wirecard、希腊主权——均在违约前数月仍被评为投资级）。

### 两大理论基础

| 理论 | 含义 | 引擎实现 |
|---|---|---|
| **马赛克理论** | 单个公共数据片段孤立无意义；组合起来形成完整图景 | 多源数据聚合、信号叠加、置信度加权 |
| **信息完备性理论** | 数据缺口不是缺陷——它们是风险信号。"我们没有这个数据"本身就告诉用户该维度存在不确定性 | 每个分析结论都附带数据完备性评分和缺口列表 |

### 架构概览

```
系统智能层（第四层）
  传染矩阵 x 集中度仪表盘 x 系统性风险指数 (SRI)
                        |
                  单一发行人结果
                        |
              ┌─────────┴─────────┐
              │   马赛克引擎      │   信号提取 + 组装 + 完备性

              └─────────┬─────────┘

           ┌────────────┼────────────┐

```

## 引擎架构

### 第一层：马赛克引擎

**从碎片化公共数据中提取信号。**

马赛克引擎作为整个分析流水线的数据入口层。它接收非结构化的多源公共数据，并将其转化为带有置信度权重的结构化信号，供下游双轨分析使用。

#### 信号提取

引擎从七类公共来源收集数据——宏观政策、行业数据、供应链价格、企业备案、诉讼/监管记录、债券市场数据和宏观经济指标——全部通过免费公开渠道（WebSearch、SEC EDGAR、央行门户、FRED、TRACE 等）。

#### 马赛克组装

单个数据点通过规则引擎按行业金字塔结构进行组装。组装过程按来源可靠性、时间新鲜度和跨来源印证性对信号进行堆叠，为每个发行人生成结构化的信号地图。

#### 完备性评估

每个分析结论都包含一个量化的**完备性评分**（0-100）和一个明确的缺口列表。引擎区分：

- **已知已知**——从多个来源确认的数据点
- **已知未知**——已识别并标记为风险信号的数据缺口
- **未知未知**——作为方法论局限性记录的结构性盲点

#### 关键指标

| 指标 | 描述 |
|---|---|
| 完备性评分 | 每个分析维度 0-100 分 |
| 信号置信度 | 每个提取信号按来源加权的置信度 |
| 缺口影响 | 每个数据缺口对评级可靠性的定性影响评估 |

### 第二层：双轨引擎

**基本面分析（Track A）与市场定价（Track B）交叉验证。**

双轨引擎并行运行两条独立分析轨道，然后对其输出进行交叉验证。轨道之间的分歧产生引擎最有价值的洞见。

#### Track A：基本面分析

应用行业金字塔框架——一个十维度评分系统（D1-D10），评估：

| 维度 | 焦点 |
|---|---|
| D1-D3 | 结构性行业地位、政策环境、竞争壁垒 |
| D4-D6 | 商业模式韧性、收入稳定性、成本结构 |
| D7-D8 | 财务政策、资本结构、流动性 |
| D9-D10 | 治理、管理层履历、外部支持 |

评分遵循严格的逐层递进：第 1 层（最重的结构性因素）必须通过，第 2 层分析才有意义；各层不可跳过。财务层（L4）作为上层判断的验证层。

#### Track B：市场定价信号

分析四个层级的市场隐含信号：

| 信号层级 | 指标 |
|---|---|
| 信用利差 | Z-spread、资产互换利差、CDS 溢价 |
| 波动率 | 价格波动率、隐含波动率偏斜 |
| 资金流 | 一级市场需求、二级换手率、投资者结构 |
| 评级迁移 | 外部评级趋势、展望变化、观察名单条目 |

#### 交叉验证矩阵

| Track A / Track B | 正面信号 | 负面信号 |
|---|---|---|
| **正面** | **相互强化**——高置信度 | **Track B 领先**——市场定价领先于基本面 |
| **负面** | **Track A 领先**——基本面先于市场定价恶化 | **相互确认**——困境的高确定性 |

冲突时，**Track A（可审计的公共财务事实）优先于**外部评级。

#### 评级映射

引擎的内部 0-10 综合评分以 18 档刻度映射三大国际评级机构——**分数越高评级越高**（9.5-10.0 = AAA 为顶端，0-0.9 = D 为底端），一票否决触发时综合评级上限锁定为 CCC。与 S&P / Moody's / Fitch 的逐档对齐以 `dev/engine/dual-track-methodology.md` §6 为唯一来源，此处不再复制。

**财务框架**：支持 IFRS（国际财务报告准则）和 US GAAP（美国公认会计原则），引擎可从发行人备案中自动检测报告标准，并针对关键差异进行调整（经营租赁资本化、研发资本化、递延税项确认）。

### 第三层：多利益相关方引擎

**六种买方角色及跨角色分析矩阵。**

不同的市场参与者通过不同视角看待同一信用。多利益相关方引擎并行运行六种买方角色的评估，然后构建一个突出共识和分歧的交叉比较矩阵。

#### 六种角色

| # | 角色 | 核心决策 | 时间视野 | 关键数据需求 |
|---|---|---|---|---|
| 1 | **信贷选择人** | "这笔信用该入账吗？"——单一发行人评级、违约概率 | 12-36 个月 | 行业金字塔、财务深潜、LGD/回收、外部支持 |
| 2 | **投资组合经理** | "这是最佳风险/回报吗？"——相对价值、行业配置 | 6-24 个月 | 相对价值指标、比较分析、曲线定位 |
| 3 | **风险官** | "集中度/传染热点在哪里？"——组合风险监控 | 持续（月度 SRI + 事件驱动） | 集中度仪表盘、传染矩阵、SRI、压力测试 |
| 4 | **交易员** | "今天是行动日吗？"——执行、市场时机 | 日内到 2 周 | L0 信号卡、实时利差、流动性状况 |
| 5 | **顾问** | "我的客户该怎么做？"——配置建议、适当性 | 3-12 个月 | 客户风险画像叠加、L1 快照、主题观点 |
| 6 | **个人投资者** | "我该持有这只债券吗？"——个人投资决策 | 6-36 个月 | 简化 L0/L1、买入/持有/卖出信号、通俗风险摘要 |

#### 跨角色矩阵

当多个利益相关方分析同一发行人时，引擎构建一个共识/分歧矩阵：

| 方面 | 共识情景 | 分歧情景 |
|---|---|---|
| 信用质量 | 所有角色一致认可评级方向 | 信贷选择人看多、风险官看空 -> 深入调查 |
| 风险偏好 | 投资组合经理和风险官在集中度上一致 | 交易员看到短期机会、风险官警告尾部风险 |
| 时间视野 | 所有角色的信号指向同一触发窗口 | 时间视野分歧暴露期限错配风险 |

该跨角色矩阵在华晨汽车案例研究中得到验证，单角色分析遗漏的结构性弱点被多角色视角揭示。

### 第四层：系统智能层

**跨发行人系统性风险感知——传染、集中度和预警。**

系统智能层是最顶层的聚合层，负责检测单一发行人分析无法揭示的系统性风险模式。它由三个集成模块和一个编码监控引擎组成。

#### 传染矩阵（19 个国际行业）

基于全球行业分类标准（GICS）的完整 19x19 跨行业传染强度矩阵，覆盖所有主要国际经济部门：

| # | 行业 | 主范式 | 传染角色 |
|---|---|---|---|
| 1 | 能源（石油与天然气） | P1：政策驱动 | 超级传播者（高向外传染） |
| 2 | 化工 | P1：政策驱动 | 中等传播者 |
| 3 | 金属与采矿 | P3：零和游戏 | 周期性放大器 |
| 4 | 建筑材料 | P4：资产租赁 | 基础设施关联传播者 |
| 5 | 资本品 | P2：技术壁垒 | 制造业传染枢纽 |
| 6 | 商业服务 | P6：网络+流量 | 低系统性关联 |
| 7 | 交通运输 | P4：资产租赁 | 物流传播载体 |
| 8 | 汽车 | P3：零和游戏 | 消费-工业桥梁 |
| 9 | 耐用消费品 | P5：品牌+渠道 | 需求周期接收者 |
| 10 | 日常消费品 | P5：品牌+渠道 | 防御型，低传染 |
| 11 | 零售 | P6：网络+流量 | 终端需求传播接收者 |
| 12 | 技术硬件（半导体） | P2：技术壁垒 | 地缘政治传染超级传播者 |
| 13 | 软件与服务 | P2：技术壁垒 | 低直接传染，高叙事溢出 |
| 14 | 生物技术与制药 | P2：技术壁垒 | 监管冲击接收者 |
| 15 | 医疗设备 | P2：技术壁垒 | 低周期传染 |
| 16 | 公用事业（受监管） | P4：资产租赁 | 防御型，低传染 |
| 17 | 电信 | P4：资产租赁 | 基础设施传染接收者 |
| 18 | 金融（银行/保险） | P1：政策驱动 | **系统性超级传播者**（最高向外传染） |
| 19 | 主权与政府支持实体 | P1：政策驱动 | 基础风险因素 |

关键衍生指标包括传染前向系数（CFC）、传染脆弱性系数（CVC）和传染净敞口比率（CNER），以及用于特定因素强度跳跃的压力升级表。

#### 五维集中度仪表盘

评估五个独立维度的组合集中度风险：

| 维度 | 评估内容 |
|---|---|
| 行业集中度 | 按 GICS 行业的 CR3 / CR5 / HHI / MAX1 |
| 区域集中度 | 单一国家/区域占比 + 弱区域占比 |
| 评级集中度 | 外部 AAA 占比 + 伪高评级占比 |
| 期限集中度 | 12 个月到期占比 + 单月峰值 |
| 融资渠道集中度 | 最大融资渠道占比 + 收缩状态 |

各维度的阈值、信号灯分档与评级调整映射以 `dev/engine/concentration-framework.md` 为唯一来源，此处不再复制。

#### 系统性风险指数（SRI）与四档温度计

SRI 将多源信号聚合为单一系统性风险读数，通过四档温度计可视化：

| 温度计档位 | SRI 区间（0-3+ 刻度） | 含义 |
|---|---|---|
| 🟢 正常 | 低于 0.5 | 系统性风险在正常范围内 |
| 🟡 关注 | 0.5 - 1.0 | 特定行业风险上升 |
| 🟠 警戒 | 1.0 - 1.8 | 风险广泛累积 |
| 🔴 危险 | 1.8 及以上 | 系统性压力迫在眉睫 |

SRI 使用连续 0-3+ 刻度——绝不使用百分比制。档位定义与规定动作以 `dev/engine/systemic-warning-framework.md` §3 为唯一来源。SRI 计算引擎（`src/sri_calculator.py`）实现聚合算法，温度计档位输入 L0 信号卡，并触发四阶段流水线的升级。

#### 展望监控引擎

第四个编码引擎（`src/outlook_engine.py`）提供：

- **12-24 个月评级展望**评估，附带方向性概率
- **90 天观察名单**，带自动触发条件
- **评级迁移矩阵**，附带历史转移概率
- **持续监控触发**，通过工作路径注册表传播

## 四阶段流水线

每一次信用分析都通过一个四阶段链式流水线进行，以 `path_id` 作为各阶段连接键：

```
┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│ ① 入口   │ -> │ ② 分析   │ -> │ ③ 报告   │ -> │ ④ QA    │
│ (路由器)  │    │ (引擎)   │    │ (构建器)  │    │ (验证器) │
└──────────┘    └──────────┘    └──────────┘    └──────────┘
```

| 阶段 | 名称 | 产物 | 宿主技能 | 状态 |
|---|---|---|---|---|
| S1 | **入口** | 工作路径单 | `credit-analysis-router` | ✅ 已交付 |
| S2 | **分析** | 分析输出 | `fixed-income-credit-analysis` | ✅ 已交付 |
| S3 | **报告** | 交付单 | `credit-report-builder` | ✅ 已交付 |
| S4 | **QA** | QA 裁定 | `credit-qa-verifier` | ✅ 已交付 |

**S1 — 入口（路由器）**：`credit-analysis-router` 技能使用四问题路由机制将模糊的用户请求（"分析这家公司"、"我该运行什么分析"）分类为具体的工作路径单。该单携带 path_id、引擎阅读顺序、模板选择和质量门规格——全部源自 `dev/engine/work-path-registry.md` 的唯一真理源。

**S2 — 分析**：`fixed-income-credit-analysis` 技能按路径单的引擎阅读顺序执行分析。对于四条有线路径，编排器（`src/pipeline.py`）直接调用相应的编码引擎：
- **WP-RO-01** -> `src/concentration_scorer.py`（五维集中度）
- **WP-RO-02** -> `src/contagion_engine.py`（传染矩阵）
- **WP-RO-03** -> `src/sri_calculator.py`（系统性风险指数）
- **WP-X-05** -> `src/outlook_engine.py`（展望监控）

所有非有线路径由 LLM 按引擎文档编排。

**S3 — 报告**：`credit-report-builder` 技能将完成的分析组装为可交付的 HTML 报告。模板选择（Type 1-18）遵循路径单规格，并映射到 L0/L1/L2 三层输出系统：
- **L0**（信号卡）——5 秒摘要：评级、展望、当日关键信号
- **L1**（快照）——一页仪表盘，含雷达图和关键异常列表
- **L2**（深潜）——完整分析报告，含金字塔分层和交叉验证

**S4 — QA**：`credit-qa-verifier` 技能执行交付前质量门审查，强制执行信号密度规则、一票否决上限合规、Mode B 护栏和唯一真理源完整性。这是四阶段链的终端阶段——未经 QA 通过的报告不得交付。

**可执行编排器**：`src/pipeline.py` 在代码中驱动整个四阶段链。它从 `dev/engine/pipeline-contract.md` 读取阶段定义（从不硬编码阶段名称），使用 `src/path_sheet.py` 验证路径单，并仅对明确有线路径调用编码引擎。所有四个产物（路径单、分析产物、交付单、QA 裁定）及其链接边的唯一真理源是 `dev/engine/pipeline-contract.md`。

## 国际范式与工作路径

### 六大国际范式（P1-P6）

引擎将所有行业分类为六种分析范式，各有不同的权重模板、评分优先级和因素侧重：

| 范式 | 代码 | 核心行业 | 关键区分因素 |
|---|---|---|---|
| **周期性** | P1 | 能源（石油与天然气）、化工、金融、主权与 GSE | 政策和商品周期决定需求天花板 |
| **防御型** | P2 | 资本品、技术硬件、软件、生物技术与制药、医疗设备 | 研发强度和知识产权创造持久壁垒；收入前估值是标准做法 |
| **成长** | P3 | 金属与采矿、汽车 | 零和竞争动态；价格竞争侵蚀利润率 |
| **受监管公用事业** | P4 | 建筑材料、交通运输、公用事业、电信 | 资产租赁特征；NOI/DSCR 是核心指标；基础设施融资 |
| **金融** | P5 | 耐用消费品、日常消费品 | 品牌权益和分销网络驱动价值；需求模式稳定 |
| **主权关联** | P6 | 商业服务、零售 | 网络效应和平台经济；规模带来递增回报 |

## 快速开始

**关键前提**：这些技能**不是**自包含的——运行时它们从**包根目录**读取 `engine/` 和 `templates/`（唯一真理源，从不复制）。安装单元是整个包根目录；**将包根目录作为项目打开**，一切即可零复制解析。

### A. npx（推荐）

```bash
npx github:tywinlu1988/credence-global
```

从 GitHub Releases 下载最新发布 zip，校验 SHA-256 后解压到 `./credence/`；用你的 agent CLI 打开该文件夹。可用 `--tag vX.Y.Z` 固定版本。

### B. GitHub Release

从 [Releases 页面](https://github.com/tywinlu1988/Credence-Global/releases) 下载最新的 `vX.Y.Z-release.zip`，对照附带的 `vX.Y.Z-release.zip.sha256` 校验，解压后打开包根目录作为项目。

### C. 克隆源码

```bash
git clone git@github.com:tywinlu1988/Credence-Global.git
cd credence-global
pip install -e .
```

### D. 运行测试

```bash
python -m pytest tests/ -q          # 完整回归套件
python scripts/consistency_check.py  # 跨文档一致性验证
```

### 第一步

1. 在 agent CLI 中打开包根目录
2. 开始对话：*"分析 XYZ 公司，半导体行业"*
3. `credit-analysis-router` 技能将你的请求路由到工作路径
4. 执行遵循四阶段流水线：入口 -> 分析 -> 报告 -> QA
5. 生成 Type-1 深潜 HTML 报告或 Type-6 评级摘要

详细的引擎文档请参见 `dev/engine/engine-overview.md`。

## 常见问题

### Q1: Credence 与传统信用分析有何不同？

Credence 解决了传统财务分析的三个系统性失效：（1）最关键的信用因素（政策、知识产权、租赁质量）永远不在资产负债表上，（2）外部评级平均延迟真实恶化 17 个月以上，（3）单视角分析遗漏结构性弱点。Credence 使用分层金字塔框架、双轨交叉验证、马赛克数据组装和多利益相关方视角来揭示传统分析遗漏的内容。

### Q2: 我需要付费订阅或 API 密钥吗？

不需要。Credence 使用零成本公共数据源——SEC EDGAR、FRED、央行门户、TRACE、行业协会报告和网络搜索。POC 阶段特意限制在零成本数据源，以证明仅凭公共数据进行有效信用分析是可行的。

### Q3: 引擎与哪些信用评级机构对齐？

引擎的内部评级等级映射到 S&P（AAA 至 D）、Moody's（Aaa 至 C）和 Fitch（AAA 至 D）。完整对齐表见 `dev/engine/dual-track-methodology.md`。

### Q4: Credence 可以与任何 AI agent CLI 一起使用吗？

可以。Credence 与 CLI 无关。它适用于 Claude Code（完全自动发现）、Codex、Cursor、Gemini 和 OpenCode。仓库根部的 `AGENTS.md` 文件是通用入口点。

### Q5: 16 条工作路径如何组织？

路径按买方角色组织：信贷选择人（2 条）、投资组合经理（2 条）、风险官（4 条）、交易员（1 条）、顾问（1 条）、个人投资者（1 条）和元/特殊用途（5 条）。每条路径指定引擎序列、报告模板和质量门。8 条路径完全活跃，6 条部分实现，2 条计划中。

### Q6: 六大国际范式是什么？

周期性（P1）、防御型（P2）、成长（P3）、受监管公用事业（P4）、金融（P5）和主权关联（P6）。每种范式决定行业分析的权重模板和评分优先级。

### Q7: "系统智能层"是什么意思？

是最顶层的聚合层，超越单一发行人分析以检测系统模式：跨行业传染（19x19 矩阵）、组合集中度（5 维）和系统性风险指数（SRI）带四档温度计。这些模块共同回答"整个组合/市场正在发生什么"，而非"这个单一发行人是否有风险"。

### Q8: 测试与 CI 如何设置？

pytest 回归套件覆盖编码引擎、流水线编排器、文档注册表、一致性检查器与发布包构建器。CI 在 Python 3.11 和 3.12 上通过 GitHub Actions 运行。本地运行使用 `python -m pytest tests/ -q`。跨文档一致性通过 `python scripts/consistency_check.py` 单独验证。

### Q9: 支持哪些财务报告标准？

IFRS（国际财务报告准则）和 US GAAP（美国公认会计原则）。引擎自动从发行人备案中检测报告标准，并针对关键差异进行调整。

### Q10: 如何开始使用四个技能链？

在 CLI 中打开包根目录，然后说"分析 [X] 公司，[Y] 行业"。`credit-analysis-router` 技能处理路由。如需直接使用技能，按 `AGENTS.md` 指令加载相关 `SKILL.md`。

### Q11: 编码引擎和 LLM 编排路径有什么区别？

四条工作路径有专门的 Python 实现（编码引擎）：集中度评分、传染矩阵、SRI 计算和展望监控。当路径 ID 匹配时，编排器（`src/pipeline.py`）直接调用这些引擎。所有其他路径由 LLM 使用引擎文档作为参考进行编排——没有 Python 代码重复方法论。

### Q12: 我可以将 Credence 用于商业目的吗？

商业使用须事先获得版权所有者的书面许可。完整条款见 [LICENSE](LICENSE) 文件。引擎以源码可用形式提供，用于非商业评估、研究和内部评估。
